# M5.2 Subagent Dark Mode Backlog

## Goal

Close the gap between “subagent defaults to off” and “subagent is actually dark-launched unless an operator enables it explicitly.”

## Delivered Scope

- gate `SubagentRunner` registration on `subagent.enabled=true`
- reject `subagent` dispatch from `swarm run` when the runner is disabled
- reject subagent follow-up injection from `swarm session follow-up` when the runner is disabled
- keep historical subagent run/session read paths intact for status, cancel, and reporting flows
- mark subagent as experimental in doctor, status, and workflow reports
- require explicit opt-in config in compatibility e2e paths that still use subagent
- add a dedicated dark-mode e2e that asserts disabled-path rejection

## Verification

- `vitest run test/unit` -> 52 files, 296 tests passed
- `vitest run test/e2e` -> 18 files, 25 tests passed
- `npm run build` -> clean

## Residual Notes

- subagent still depends on the bridge-backed adapter path because a stable public spawn export is not available yet
- historical workflows that already contain subagent tasks remain readable, but new dispatch is intentionally blocked unless operators opt in

## M5.3 Carryover

- narrow ACP bridge usage to explicit fallback only
- keep new ACP work on the public control-plane path first
- improve operator diagnostics around “public path selected” vs “bridge fallback selected”
