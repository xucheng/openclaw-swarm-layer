# M3.0 Session Foundation

## Goal

Establish the first session-aware orchestration layer without jumping directly into full persistent-session reuse.

`M3.0` is the schema, state, and operator foundation for reusable sessions.
It is not yet the milestone where the system fully reuses long-lived ACP sessions across many tasks.

## Outcome

After `M3.0`, the plugin should be able to:

- describe session intent at the task level
- persist session records independently from run records
- track session lifecycle across runs
- expose session inventory and inspection to operators
- prepare runner logic for future reusable-session selection

It should not yet claim:

- full persistent ACP session reuse across arbitrary tasks
- thread-bound follow-up workflows
- recovery-heavy session resurrection after restart

## Scope

### In Scope

- session-aware schema additions
- session registry in local swarm state
- task-level session policy metadata
- session lifecycle state model
- session-aware operator CLI for listing and inspecting sessions
- status/report visibility for sessions
- migration from current `M2.x` state to `M3.0` state layout

### Out Of Scope

- production-grade persistent session reuse
- thread binding
- steer/follow-up commands
- advanced retry and recovery
- distributed session coordination

## Key Design Decisions

### 1. Session Is A First-Class Record

Add a dedicated session registry instead of inferring everything from `RunRecord`.

Recommended persisted entity:

```ts
type SessionRecord = {
  sessionId: string;
  runner: "acp" | "subagent";
  projectRoot: string;
  scope: {
    specId?: string;
    bindingKey?: string;
    taskKind?: string;
  };
  mode: "oneshot" | "persistent";
  state: "created" | "active" | "idle" | "closed" | "failed" | "orphaned";
  createdAt: string;
  updatedAt: string;
  lastRunId?: string;
  lastTaskId?: string;
  providerRef: {
    sessionKey?: string;
    backend?: string;
    backendSessionId?: string;
    agentSessionId?: string;
  };
  summary?: string;
  metadata?: Record<string, unknown>;
};
```

### 2. Task Carries Session Intent

Extend `TaskNode` with session policy, but do not force immediate reuse semantics in `M3.0`.

Recommended shape:

```ts
type TaskSessionPolicy = {
  policy: "none" | "create_persistent" | "reuse_if_available" | "require_existing";
  bindingKey?: string;
  preferredSessionId?: string;
};
```

### 3. Session Lifecycle Must Be Separate From Run Lifecycle

`RunRecord` answers: what happened during one execution.

`SessionRecord` answers: what long-lived channel/context exists for future execution.

That separation is the core foundation for `M3.1+`.

## State Layout Changes

Current state layout already contains:

```text
<repo>/.openclaw/swarm/
├── workflow-state.json
├── specs/
├── runs/
├── reports/
```

`M3.0` should add:

```text
<repo>/.openclaw/swarm/
├── sessions/
│   ├── <sessionId>.json
│   └── ...
```

No need for a separate session index file if lookup by directory scan remains cheap enough.

## CLI Surface For M3.0

Recommended additions:

- `openclaw swarm session list --project <path>`
- `openclaw swarm session inspect --project <path> --session <id>`

Recommended non-additions yet:

- `session reuse`
- `session resume`
- `session steer`

Those belong to `M3.1+`.

## Acceptance Criteria

`M3.0` should be considered complete when:

- session schema exists and validates
- session records persist independently from runs
- session lifecycle can be derived and updated from execution outcomes
- `swarm status` includes session summary
- operator can list and inspect sessions
- current `M2.x` state migrates cleanly forward
- unit and e2e coverage exist for session registry basics

## Risks To Manage

- over-designing reuse before actual `M3.1`
- coupling `SessionRecord` too tightly to ACP internals only
- creating session state that cannot support subagent and ACP together
- letting status/report changes leak future policy prematurely

## Recommended Implementation Order

1. schema and types
2. session store and migration
3. session lifecycle mapping
4. CLI list/inspect
5. status/report integration
6. tests and e2e regression
